# Задача 1. BPMN 2.0 модель процесса выдачи ИТ-оборудования

## Описание процесса

В рамках задачи была построена модель процесса выдачи ИТ-оборудования пользователю по заявке.

В процессе участвуют следующие роли:

- пользователь офиса;
- первая линия технической поддержки;
- отдел снабжения;
- логистика;
- поставщик.

Также в процессе используются следующие информационные системы:

- **система №1** - система регистрации обращений пользователей;
- **система №2** - система заявок на оборудование;
- **система №3** - система учёта оборудования на складе.

Основной процесс начинается с того, что пользователь обращается в первую линию технической поддержки по почте или телефону. Первая линия регистрирует обращение в системе №1, выдаёт пользователю номер обращения и создаёт заявку на оборудование в системе №2 для отдела снабжения.

Отдел снабжения получает уведомление о заявке по почте и в личном кабинете системы №2, принимает заявку и проверяет наличие оборудования в системе №3.

Если оборудование есть в наличии, снабжение вызывает логистику, передаёт оборудование, логистика доставляет его в первую линию технической поддержки. После этого техническая поддержка выдаёт оборудование пользователю и закрывает заявки в системах №1 и №2.

Если оборудования нет, отдел снабжения отправляет заявку на закупку поставщику по почте, ожидает поставку, принимает оборудование в системе №3, вызывает логистику и далее процесс идёт по той же схеме: оборудование передаётся в техническую поддержку, выдаётся пользователю, а заявки закрываются.

После получения оборудования пользователь может выставить оценку сервиса.

## Вопросы и замечания к процессу

В описании процесса были выявлены следующие неоднозначности:

1. Не указано явно, кто создаёт заявку в системе №1. Предполагается, что это делает первая линия технической поддержки.
2. Не указано, из какой именно системы пользователь получает номер обращения. Предполагается, что номер формируется в системе №1.
3. Не описана связь между заявкой в системе №1 и заявкой в системе №2.
4. Не указаны статусы заявки при ожидании закупки или поставки оборудования.
5. Не описаны негативные сценарии: отказ поставщика, задержка поставки, брак оборудования, отказ пользователя.
6. Неясно, резервируется ли оборудование в системе №3 после проверки наличия.
7. Не указано, является ли оценка сервиса обязательной или необязательной.

# Задача 2. Требования к функциональности публикации товара на маркетплейсе

## User Story

**Как продавец**, я хочу иметь возможность создать карточку товара в личном кабинете и опубликовать её на маркетплейсе, чтобы покупатели могли увидеть мой товар на витрине, ознакомиться с его описанием, ценой, фотографиями и оформить покупку.

## Основные требования

Продавец должен иметь возможность:

- создать новую карточку товара;
- заполнить название, описание, категорию, цену, остатки, характеристики и фотографии;
- сохранить товар как черновик;
- отправить товар на публикацию;
- отредактировать товар;
- снять товар с публикации.

Система должна:

- проверять обязательные поля;
- проверять корректность цены и остатков;
- проверять наличие хотя бы одной фотографии;
- показывать ошибки заполнения;
- менять статус товара;
- отправлять товар на модерацию или публиковать его;
- отображать опубликованный товар на витрине маркетплейса.

## Статусы товара

В рамках процесса публикации товара можно выделить следующие статусы:

- `Черновик`;
- `На модерации`;
- `Опубликован`;
- `Отклонён`;
- `Снят с публикации`.

## Use Case 1. Создание карточки товара

| Поле | Описание |
|---|---|
| Название | Создание карточки товара |
| Актор | Продавец |
| Цель | Создать карточку товара в личном кабинете |
| Предусловие | Продавец авторизован |
| Постусловие | Карточка товара создана в статусе “Черновик” |

### Основной сценарий

1. Продавец открывает личный кабинет.
2. Переходит в раздел “Товары”.
3. Нажимает “Добавить товар”.
4. Система открывает форму создания товара.
5. Продавец заполняет данные товара.
6. Продавец нажимает “Сохранить”.
7. Система проверяет данные.
8. Система сохраняет карточку товара в статусе “Черновик”.

### Альтернативный сценарий

Если продавец не заполнил обязательные поля, система подсвечивает ошибки и предлагает исправить данные.

## Use Case 2. Загрузка фотографий товара

| Поле | Описание |
|---|---|
| Название | Загрузка фотографий товара |
| Актор | Продавец |
| Цель | Добавить изображения товара в карточку |
| Предусловие | Открыта форма создания или редактирования товара |
| Постусловие | Фотографии добавлены в карточку товара |

### Основной сценарий

1. Продавец нажимает “Загрузить фото”.
2. Выбирает изображения на устройстве.
3. Система проверяет формат и размер файлов.
4. Система загружает изображения.
5. Система отображает превью фотографий.
6. Продавец может изменить порядок фотографий.
7. Продавец сохраняет карточку.

### Ограничения

- Допустимые форматы: JPG, JPEG, PNG, WebP.
- Максимальный размер одного файла: 10 МБ.
- Минимальное количество фотографий для публикации: 1.
- Максимальное количество фотографий: 10.

## Use Case 3. Публикация товара

| Поле | Описание |
|---|---|
| Название | Публикация товара |
| Актор | Продавец |
| Цель | Опубликовать товар на витрине маркетплейса |
| Предусловие | Карточка товара создана |
| Постусловие | Товар опубликован или отправлен на модерацию |

### Основной сценарий

1. Продавец открывает карточку товара.
2. Нажимает кнопку “Опубликовать”.
3. Система проверяет заполнение обязательных полей.
4. Система проверяет цену, остатки, категорию, описание и фотографии.
5. Если данные корректны, товар получает статус “На модерации” или “Опубликован”.
6. Если данные некорректны, система показывает список ошибок.
7. После успешной публикации товар отображается на витрине маркетплейса.

## Use Case 4. Модерация товара

| Поле | Описание |
|---|---|
| Название | Модерация товара |
| Актор | Модератор / система модерации |
| Цель | Проверить соответствие товара правилам маркетплейса |
| Предусловие | Товар находится в статусе “На модерации” |
| Постусловие | Товар опубликован или отклонён |

### Основной сценарий

1. Система или модератор получает карточку товара.
2. Выполняется проверка названия, описания, фото, категории и характеристик.
3. Если нарушений нет, товар получает статус “Опубликован”.
4. Если есть нарушения, товар получает статус “Отклонён”.
5. Продавец получает уведомление о результате.

# Задача 3. REST API регистрации пользователя

## Допущения

Так как изображение интерфейса регистрации недоступно для просмотра, в решении были приняты следующие допущения: форма регистрации содержит поля:

- `username`;
- `email`;
- `password`;
- `confirmPassword`.

Если в реальном интерфейсе есть дополнительные поля, например `firstName`, `lastName`, `phone`, их можно добавить в тело запроса по аналогии.

---

## Задача 3.1. Описание REST API

Для регистрации пользователя используется следующий метод:

```http
POST /api/v1/users/register
Content-Type: application/json
```

Назначение метода: создать нового пользователя в системе после нажатия кнопки **Register**.

## Входные параметры

| Параметр | Тип | Обязательность | Ограничения |
|---|---|---|---|
| `username` | string | да | 3–30 символов, латинские буквы, цифры, `_`, `-`, `.` |
| `email` | string | да | Валидный email, максимум 254 символа, уникальный |
| `password` | string | да | 8–64 символа, минимум одна буква и одна цифра |
| `confirmPassword` | string | да | Должен совпадать с `password` |

## Пример запроса

```json
{
  "username": "alex_ivanov",
  "email": "alex.ivanov@example.com",
  "password": "Password123",
  "confirmPassword": "Password123"
}
```

## Успешный ответ

HTTP-код:

```http
201 Created
```

Пример ответа:

```json
{
  "userId": "0f9b7a0e-7e6a-4f58-b91f-4b77f7c82d6e",
  "username": "alex_ivanov",
  "email": "alex.ivanov@example.com",
  "status": "ACTIVE",
  "createdAt": "2026-05-03T10:30:00Z"
}
```

## Выходные параметры при успешном ответе

| Параметр | Тип | Обязательность | Ограничения |
|---|---|---|---|
| `userId` | string / UUID | да | Уникальный идентификатор пользователя |
| `username` | string | да | Логин пользователя |
| `email` | string | да | Email пользователя |
| `status` | string | да | Например: `ACTIVE` |
| `createdAt` | string / datetime | да | Дата и время создания пользователя в ISO 8601 |

## Ошибки

| HTTP-код | Ошибка | Описание |
|---|---|---|
| `400 Bad Request` | `VALIDATION_ERROR` | Ошибка валидации входных данных |
| `400 Bad Request` | `INVALID_JSON` | Некорректный JSON в теле запроса |
| `409 Conflict` | `USER_ALREADY_EXISTS` | Пользователь с таким email уже существует |
| `409 Conflict` | `USERNAME_ALREADY_EXISTS` | Пользователь с таким username уже существует |
| `500 Internal Server Error` | `INTERNAL_SERVER_ERROR` | Внутренняя ошибка сервера |

## Пример ошибки валидации

HTTP-код:

```http
400 Bad Request
```

Тело ответа:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation error",
  "errors": [
    {
      "field": "email",
      "message": "Email has invalid format"
    },
    {
      "field": "password",
      "message": "Password must contain at least 8 characters"
    }
  ]
}
```

## Пример ошибки при существующем пользователе

HTTP-код:

```http
409 Conflict
```

Тело ответа:

```json
{
  "code": "USER_ALREADY_EXISTS",
  "message": "User with this email already exists"
}
```

## OpenAPI YAML

```yaml
openapi: 3.0.3
info:
  title: User Registration API
  version: 1.0.0

paths:
  /api/v1/users/register:
    post:
      summary: Register new user
      description: Creates a new user account after clicking the Register button.
      operationId: registerUser
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required:
                - username
                - email
                - password
                - confirmPassword
              properties:
                username:
                  type: string
                  minLength: 3
                  maxLength: 30
                  pattern: "^[a-zA-Z0-9_.-]+$"
                  example: "alex_ivanov"
                email:
                  type: string
                  format: email
                  maxLength: 254
                  example: "alex.ivanov@example.com"
                password:
                  type: string
                  format: password
                  minLength: 8
                  maxLength: 64
                  example: "Password123"
                confirmPassword:
                  type: string
                  format: password
                  minLength: 8
                  maxLength: 64
                  example: "Password123"

      responses:
        "201":
          description: User successfully created
          content:
            application/json:
              schema:
                type: object
                required:
                  - userId
                  - username
                  - email
                  - status
                  - createdAt
                properties:
                  userId:
                    type: string
                    format: uuid
                    example: "0f9b7a0e-7e6a-4f58-b91f-4b77f7c82d6e"
                  username:
                    type: string
                    example: "alex_ivanov"
                  email:
                    type: string
                    format: email
                    example: "alex.ivanov@example.com"
                  status:
                    type: string
                    example: "ACTIVE"
                  createdAt:
                    type: string
                    format: date-time
                    example: "2026-05-03T10:30:00Z"

        "400":
          description: Validation error or invalid JSON
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/ErrorResponse"

        "409":
          description: User already exists
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/ErrorResponse"

        "500":
          description: Internal server error
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/ErrorResponse"

components:
  schemas:
    ErrorResponse:
      type: object
      required:
        - code
        - message
      properties:
        code:
          type: string
          example: "VALIDATION_ERROR"
        message:
          type: string
          example: "Validation error"
        errors:
          type: array
          items:
            type: object
            required:
              - field
              - message
            properties:
              field:
                type: string
                example: "email"
              message:
                type: string
                example: "Email has invalid format"
```

# Задача 3.2. Алгоритм создания пользователя на стороне бэкенда

После получения запроса на регистрацию бэкенд выполняет следующие действия:

1. Получает запрос `POST /api/v1/users/register`.
2. Проверяет, что `Content-Type` равен `application/json`.
3. Проверяет корректность JSON.
4. Извлекает поля `username`, `email`, `password`, `confirmPassword`.
5. Нормализует данные: удаляет пробелы в начале и конце, email приводит к нижнему регистру.
6. Проверяет заполнение обязательных полей.
7. Проверяет корректность username:
   - длина от 3 до 30 символов;
   - допустимы латинские буквы, цифры, `_`, `-`, `.`.
8. Проверяет корректность email:
   - валидный формат;
   - длина не больше 254 символов.
9. Проверяет сложность password:
   - длина от 8 до 64 символов;
   - содержит минимум одну букву;
   - содержит минимум одну цифру.
10. Проверяет совпадение `password` и `confirmPassword`.
11. Проверяет уникальность email в базе данных.
12. Проверяет уникальность username в базе данных.
13. Хэширует пароль.
14. Создаёт объект пользователя.
15. Сохраняет пользователя в базу данных.
16. Возвращает успешный ответ `201 Created`.
17. В случае ошибки возвращает соответствующий HTTP-код и сообщение.
